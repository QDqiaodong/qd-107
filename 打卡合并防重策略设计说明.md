# 打卡合并防重策略设计说明

## 一、需求背景

针对运动打卡场景，存在两类问题需要解决：

1. **误触重复提交**：用户短时间内多次点击提交按钮，导致同一打卡记录重复入库
2. **并发双发问题**：网络延迟或前端重试导致同一请求同时到达后端，防重策略失效
3. **近似内容合并**：用户提交的两条打卡内容高度相似（时长、距离接近，备注略有差异），应自动合并为一条记录

## 二、设计目标

- 避免误触造成的重复打卡记录
- 解决并发双发场景下的重复数据问题
- 对高度相似的打卡内容自动合并，保留主记录并整合统计数据
- 合并过程对用户透明，返回友好提示

## 三、核心策略

### 3.1 合并时间窗口

| 参数 | 值 | 说明 |
|------|-----|------|
| 时间窗口 | 5分钟 | 检测当前打卡往前5分钟内的记录 |
| 运动类型匹配 | 精确匹配 | 必须是同一运动类型 |
| 用户匹配 | 精确匹配 | 必须是同一用户 |

### 3.2 相似度判定规则

#### 3.2.1 时长相似度

- **绝对阈值**：时长差 ≤ 5分钟，判定为相似
- **相对阈值**：时长差 / 较大时长 ≤ 10%，判定为相似
- 满足任一条件即为相似

#### 3.2.2 距离相似度

- 距离差 / 较大距离 ≤ 10%，判定为相似
- 任一方为空或为0时，跳过距离判断

#### 3.2.3 备注相似度

- 使用 **Levenshtein 编辑距离**算法计算相似度
- 相似度阈值：**80%**
- 两端空白字符会先 trim 再比较
- 两方都为空时，判定为相似
- 仅一方为空时，判定为不相似

### 3.3 合并规则

| 字段 | 合并策略 | 说明 |
|------|----------|------|
| 时长（duration） | 取较大值 | 保留运动量更大的记录 |
| 卡路里（calorie） | 取较大值 | 保留消耗更多的记录 |
| 距离（distance） | 取较大值 | 保留距离更长的记录 |
| 计划ID（planId） | 优先保留已有值 | 新记录有而原记录没有时补充 |
| 备注（remark） | 相似则保留较长文本，不相似则拼接 | 用中文分号"；"分隔 |
| 图片（images） | 去重合并 | 逗号分隔，按Set去重 |
| 打卡时间 | 保留原记录时间 | 以最早的记录为主记录 |

## 四、并发防重方案

### 4.1 方案架构

采用 **Redis 分布式锁 + 降级查询** 的双层防护机制：

```
请求到达
   ↓
尝试获取分布式锁
   ├─ 获取成功 → 执行业务逻辑（查重+合并/新增）→ 释放锁 → 返回结果
   └─ 获取失败 → 重试3次（间隔300ms）
                  ├─ 重试成功 → 执行业务逻辑
                  └─ 重试失败 → 降级查询（直接查库是否有相似记录）
                                   ├─ 有相似记录 → 返回合并结果
                                   └─ 无相似记录 → 抛出"系统繁忙"异常
```

### 4.2 分布式锁设计

| 参数 | 值 | 说明 |
|------|-----|------|
| 锁 Key | `checkin:lock:{userId}:{sportTypeId}` | 按用户+运动类型粒度 |
| 锁过期时间 | 10秒 | 防止死锁 |
| 重试次数 | 3次 | 提高成功率 |
| 重试间隔 | 300ms | 避免频繁竞争 |
| 解锁方式 | Lua 脚本 | 保证原子性，防止误删 |

### 4.3 锁安全机制

- 使用 UUID 作为锁的 value，确保只有加锁者能解锁
- 解锁使用 Lua 脚本保证原子性
- 采用 ThreadLocal 存储当前线程的锁 value
- finally 块确保锁一定会释放

## 五、接口设计

### 5.1 打卡接口

**请求**：
```
POST /api/checkin
Content-Type: application/json

{
  "userId": 1,
  "sportTypeId": 1,
  "duration": 30,
  "distance": 5.0,
  "remark": "今天跑步很开心",
  "images": "img1.jpg,img2.jpg",
  "planId": 1
}
```

**响应（新增成功）**：
```json
{
  "code": 200,
  "message": "打卡成功",
  "data": {
    "record": { ...打卡记录对象... },
    "merged": false,
    "mergeTip": "打卡成功"
  }
}
```

**响应（合并成功）**：
```json
{
  "code": 200,
  "message": "检测到短时间内存在相似打卡，已自动合并",
  "data": {
    "record": { ...合并后的打卡记录... },
    "merged": true,
    "mergeTip": "检测到短时间内存在相似打卡，已自动合并"
  }
}
```

### 5.2 响应字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| record | Object | 打卡记录详情 |
| merged | Boolean | 是否发生了合并 |
| mergeTip | String | 提示信息，可直接展示给用户 |

### 6.3 文本相似度算法

采用 **Levenshtein 编辑距离** 算法，计算将一个字符串转换为另一个字符串所需的最少操作次数（插入、删除、替换）。

相似度计算公式：
```
相似度 = 1 - (编辑距离 / 较长字符串长度)
```

示例：
- "今天跑步很开心" vs "今天跑步很開心" → 相似度 ≈ 85.7% → 判定为相似
- "跑步30分钟" vs "跑步三十分" → 相似度 ≈ 57.1% → 判定为不相似

## 七、边界场景处理

### 7.1 并发双发

- **场景**：用户双击提交按钮，两个请求同时到达
- **处理**：Redis 锁保证只有一个请求能执行新增，另一个请求在重试后查询到已新增的记录，返回合并结果

### 7.2 Redis 不可用

- **场景**：Redis 服务宕机
- **处理**：Redis 操作异常会导致锁获取失败，触发降级逻辑，直接查询数据库是否有相似记录

### 7.3 锁过期

- **场景**：业务执行时间超过锁过期时间
- **处理**：锁过期后其他请求可进入，但由于数据库层面有合并检测，最终数据仍是正确的（可能会有合并）

### 7.4 空值处理

- 时长、距离、备注等字段为空时，相似度判断会跳过该维度
- 合并时空值不会覆盖已有值

## 九、验证要点

1. **功能验证**
   - 同一用户同一运动类型，5分钟内提交相同数据 → 应合并
   - 同一用户同一运动类型，间隔超过5分钟 → 不应合并
   - 不同运动类型 → 不应合并
   - 不同用户 → 不应合并

2. **并发验证**
   - 同时发起两个相同请求 → 最终只有一条记录
   - 并发请求返回结果中 merged 标记正确

3. **相似度验证**
   - 备注完全相同 → 合并
   - 备注有少量差异（80%以上相似） → 合并
   - 备注差异较大 → 不合并

4. **数据正确性验证**
   - 合并后时长取较大值
   - 合并后卡路里取较大值
   - 合并后距离取较大值
   - 合并后备注正确处理
   - 合并后图片去重
