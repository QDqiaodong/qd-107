# 打卡图片一致性修复说明

## 问题描述

打卡图片在移除后仍然可能保存旧图片，导致图片选择、预览、删除和记录展示不一致。用户在打卡页面删除图片后，提交的打卡记录中仍然包含已删除的图片；或者在打卡合并场景下，用户已移除的旧图片仍然出现在最终记录中。

## 根本原因分析

### 前端问题（Checkin.vue）

#### 问题1：双数据源不同步

打卡页面存在两个独立的图片数据源：
- `fileList`：Element Plus Upload 组件的文件列表，用于 UI 展示（预览、删除按钮等）
- `formData.images`：实际提交给后端的图片数据数组

两个数据源通过索引位置关联，但由于各自独立维护，容易出现索引错位、数据不一致的问题。

#### 问题2：`handleFileChange` 逻辑不完整

Element Plus 的 `on-change` 事件在以下场景都会触发：
- 文件添加
- 文件删除
- 文件状态变化（如上传成功/失败）

但原代码只处理了"文件添加"的场景（`push` 新图片），没有处理删除或替换场景。

#### 问题3：`handleFileRemove` 索引不可靠

原代码使用 `fileList.value.indexOf(file)` 查找被删除文件的索引，然后从 `formData.images` 中删除对应索引的元素。这种方式存在两个问题：
1. `indexOf` 通过引用比较，Element Plus 传入的 file 对象可能与 fileList 中的不是同一个引用，导致返回 -1
2. 删除逻辑的执行顺序存在问题，可能导致索引错位

### 后端问题（CheckinRecordServiceImpl.java）

#### 问题：`mergeImages` 采用简单并集策略

打卡合并时，`mergeImages` 方法将旧记录和新记录的图片做简单的集合并集。这意味着：
- 即使用户在新打卡中移除了某些旧图片
- 合并后这些旧图片仍然会保留在最终记录中

这种策略不符合"新打卡代表用户最新意图"的预期。

## 修复方案

### 前端修复

#### 核心思路：单一数据源原则

以 `fileList` 作为唯一的真相来源（Single Source of Truth），`formData.images` 通过同步函数从 `fileList` 派生，确保两者始终一致。

#### 具体实现

**1. 新增同步函数 `syncImagesFromFileList`**

```javascript
const syncImagesFromFileList = () => {
  formData.images = []
  fileList.value.forEach(file => {
    if (file.url) {
      formData.images.push(file.url)
    }
  })
}
```

该函数遍历 `fileList` 中的所有文件，将有 `url` 属性的文件提取到 `formData.images` 中，确保两者一一对应。

**2. 重构 `handleFileChange`**

```javascript
const handleFileChange = (file, files) => {
  fileList.value = files
  if (file.raw && !file.url) {
    const reader = new FileReader()
    reader.onload = (e) => {
      file.url = e.target.result
      syncImagesFromFileList()
    }
    reader.readAsDataURL(file.raw)
  } else {
    syncImagesFromFileList()
  }
}
```

改进点：
- 接收完整的 `files` 参数（Element Plus 的 on-change 回调提供）
- 先更新 `fileList`，再同步 `formData.images`
- 文件读取完成后再次同步，确保 base64 数据正确写入

**3. 简化 `handleFileRemove`**

```javascript
const handleFileRemove = (file, files) => {
  fileList.value = files
  syncImagesFromFileList()
}
```

改进点：
- 直接使用 Element Plus 传入的 `files`（删除后的文件列表）
- 通过同步函数统一更新 `formData.images`
- 不再依赖不可靠的 `indexOf` 索引查找

### 后端修复

#### 核心思路：新记录优先策略

打卡合并时，图片以新打卡记录为准，因为新记录代表用户最新的意图。

#### 具体实现

**修改 `mergeImages` 方法**

修复前：
```java
private String mergeImages(String img1, String img2) {
    if (!StringUtils.hasText(img1) && !StringUtils.hasText(img2)) {
        return null;
    }
    if (!StringUtils.hasText(img1)) {
        return img2;
    }
    if (!StringUtils.hasText(img2)) {
        return img1;
    }
    Set<String> imageSet = new HashSet<>();
    imageSet.addAll(Arrays.asList(img1.split(",")));
    imageSet.addAll(Arrays.asList(img2.split(",")));
    return imageSet.stream().filter(StringUtils::hasText).collect(Collectors.joining(","));
}
```

修复后：
```java
private String mergeImages(String existingImg, String newImg) {
    if (StringUtils.hasText(newImg)) {
        return newImg;
    }
    return StringUtils.hasText(existingImg) ? existingImg : null;
}
```

策略说明：
- 如果新打卡包含图片（非空）：使用新打卡的图片列表（尊重新意图，用户移除的图片不再保留）
- 如果新打卡没有图片：保留旧记录的图片（防止用户误操作丢失图片）

## 修复前后对比

| 场景 | 修复前行为 | 修复后行为 |
|------|-----------|-----------|
| 用户添加图片后删除其中一张 | 被删除的图片可能仍在 formData.images 中 | 预览和实际数据完全一致，删除即移除 |
| 用户多次添加/删除图片 | 索引可能错位，数据混乱 | 始终以 fileList 为准，同步更新 |
| 打卡合并（新打卡有不同图片） | 新旧图片取并集，旧图片保留 | 以新打卡为准，旧图片被替换 |
| 打卡合并（新打卡无图片） | 保留旧图片（行为一致） | 保留旧图片（行为一致） |

## 验证方法

### 前端验证
1. 进入打卡页面，选择多张图片
2. 删除其中 1-2 张图片
3. 观察预览区域与已删除图片是否一致
4. 提交打卡，查看返回记录中的图片数量是否与最终选择一致

### 后端验证
1. 第一次打卡，上传图片 A、B、C
2. 短时间内（5分钟内）第二次打卡，只上传图片 A、B（触发合并）
3. 检查合并后的记录，应只包含图片 A、B（不包含 C）
4. 第三次打卡，不上传图片
5. 检查合并后的记录，应保留图片 A、B（不因无图而清空）

## 设计原则

1. **单一数据源原则**：避免维护多个独立的状态副本，通过派生确保一致性
2. **最新意图优先**：新操作代表用户最新意图，应优先采用
3. **容错设计**：关键操作（如清空图片）保留安全网，防止误操作导致数据丢失
